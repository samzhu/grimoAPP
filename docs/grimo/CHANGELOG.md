### 新增

- **S016 — MVP 人工驗證閘門（Skill CLI + 投影）**（2026-04-20）。為 M5 出貨的三大能力加上人工可驗證的 CLI 入口。新增 `SkillCommandRunner`（`skills/adapter/in/cli/`）實作 `grimo skill list/enable/disable` 子命令，暴露 S012 的 `SkillRegistryUseCase` 至終端。新增 `SkillProjectionUseCase` + `SkillProjectionService`（`skills/application/service/`），在 `grimo chat` 啟動 Claude Code 前將已啟用 Skill 從 `~/.grimo/skills/` 複製至 `<workdir>/.claude/skills/`，使 Claude Code 原生載入。跨模組接線：`skills/application/port/in/` 加 `@NamedInterface("api")`，`agent` 模組 `allowedDependencies` 新增 `"skills::api"`（首次 agent → skills 依賴）。13 個測試（6 SkillCommandRunner + 4 SkillProjectionService + 3 ChatCommandRunner S016 新增）。含結構化人工驗證計畫（對話 + resume + skill 投影端對端）。詳見 `docs/grimo/specs/archive/2026-04-20-S016-mvp-manual-verification.md` 第 7 節。
- **S012 — Skill 登錄檔（SkillRegistryUseCase）**（2026-04-20）。在 `skills` 模組建立 Skill 登錄檔。重寫 `spring-ai-agent-utils` 0.7.0 的 `Skills` + `MarkdownParser`（同介面，SnakeYAML 內部），支援完整 agentskills.io YAML frontmatter（巢狀 metadata map、YAML list、多行 scalar）。掃描 `~/.grimo/skills/*/SKILL.md`，遵循 agentskills.io 開放標準驗證（name 格式、name 與目錄名一致、description 非空）。無效檔案以 WARN log 跳過不崩潰。啟用/停用狀態持久化至 `~/.grimo/skills/.state.json`（版本化 JSON）。產出型別為框架原生 `SkillsTool.Skill` record，可直接被 `SkillsFunction` → `ToolCallback` → `ChatClient` 消費。Hexagonal 分層：`SkillRegistryUseCase`（入站埠）、`SkillStorePort`（出站埠）、`SkillRegistryService`（@Service）、`FileSystemSkillStoreAdapter`（@Component，自行建構 ObjectMapper）。`allowedDependencies = { "core" }`。25 個測試（7 MarkdownParser + 2 SkillEntry + 7 SkillRegistryService mock + 9 FileSystemSkillStoreAdapter 真實 FS）。**設計偏離：** §4.6 原設計注入 ObjectMapper，實作改為無參建構子自行建構（專案無 Jackson auto-config bean）。**POC（8/8 通過）** 驗證 SnakeYAML frontmatter `---` 分隔符處理需手動擷取。詳見 `docs/grimo/specs/archive/2026-04-19-S012-skill-registry.md` 第 7 節。
- **S011 — Session Resume（grimo chat --resume）**（2026-04-20）。為 `grimo chat` 加入 `--resume` 旗標，透過 Claude CLI 原生的 `--continue` flag 接回同工作目錄下最近一次的 session。**零新依賴** — Claude CLI 自己管歷史（`~/.claude/projects/` 下的 JSONL transcript）。實作 `ClaudeSessionConnector`（same-package helper，`org.springaicommunity.agents.claude`）存取 package-private 的 `ClaudeAgentSession` 建構子，使用 `CLIOptions.builder().continueConversation(true)` 建立帶 `--continue` 的 `ClaudeSyncClient`，回傳標準 `AgentSession`。`MainAgentChatService.resumeChat()` 嘗試 resume，失敗時自動降級為新 session（印出 "No previous session found, starting new session"）。`ChatCommandRunner` 解析 `--resume` option 路由至 `resumeChat()`。**設計轉折：** v1 設計（decorator + spring-ai-session-jdbc + H2）基於「AgentSession 沒有持久化」的錯誤假設 — POC（8/8 通過）驗證 Claude CLI 的 `--resume` / `--continue` 完美運作，使得整個持久化層不必要。估算從 S(10) 降為 XS(6)。`--resume <id>` 指定 ID 恢復功能列入技術債。詳見 `docs/grimo/specs/archive/2026-04-19-S011-session-persistence.md` 第 7 節。
- **S007 — 主代理 CLI 對話（grimo chat）**（2026-04-19）。在 `agent` 模組實作 `grimo chat` 子命令，使用 agent-client 0.12.2 的 `AgentSessionRegistry` / `AgentSession` 介面維持持續運行的主機 claude CLI process。`ChatCommandRunner`（`ApplicationRunner`）偵測 `chat` 子命令，`MainAgentChatService` 驅動終端 REPL 迴圈：讀取使用者輸入 → `session.prompt()` → 印出回應。`/exit`、`/quit`、`Ctrl+D` 乾淨退出。CLI 未安裝時以 `ChatSessionException` 印出安裝指引（無堆疊追蹤）。注入型別為 `AgentSessionRegistry`（介面）而非 `ClaudeAgentSessionRegistry`（具體類別），有利 S011 decorator 透明替換。`AgentModuleConfig` 建立 `ClaudeAgentSessionRegistry` bean（timeout 30m）。8 個 unit tests 覆蓋 AC-1/2/3，1 個 IT（需本機 claude CLI）以 `assumeTrue` 控制。**里程碑 M4 完成。** 詳見 `docs/grimo/specs/archive/2026-04-18-S007-main-agent-chat.md` 第 7 節。
- **S006 — CLI 配置研究 + 策略驗證**（2026-04-18）。調查三 CLI（Claude Code、Codex CLI、Gemini CLI）的完整配置介面，成果落地於 `docs/grimo/cli-config-matrix.md`（五維對照表：認證、記憶體、遙測、設定檔、env var 覆寫，每欄附官方文件 URL）。驗證成果固化為 `CliInvocationOptions` record（`cli/api/`），含三個靜態工廠：`claude(token)` 注入 `CLAUDE_CODE_OAUTH_TOKEN` + 記憶體停用 + 遙測停用、`codex()` 設定 `CODEX_HOME` + analytics 停用、`gemini(apiKey)` 注入 `GEMINI_API_KEY` + 重導 `GEMINI_CLI_HOME`。Docker IT 驗證：Claude 認證（Keychain → accessToken 提取 ✅）、Claude 記憶體停用（`CLAUDE_CODE_DISABLE_CLAUDE_MDS=1` ✅）、Codex 認證（RO 掛載 auth.json 單檔 ✅）、缺少認證時清楚錯誤（✅）。Gemini IT 編譯通過待 API key 驗證。**Key findings：** (1) `CLAUDE_CODE_OAUTH_TOKEN` 接受 accessToken 字串而非 Keychain JSON blob；(2) Codex CLI 需 writable CODEX_HOME，只能 RO 掛載 `auth.json` 單檔；(3) `--bare` 不讀取 OAuth token 且殺 skill 載入，確認不可用。**里程碑 M3 完成。** 詳見 `docs/grimo/specs/archive/2026-04-18-S006-cli-config-validation.md` 第 7 節。
- **S005 — 容器化 AgentModel 適配器**（2026-04-18）。在 `cli` 模組建立 `ContainerizedAgentModelFactory`（`@NamedInterface("api")`），為每個容器動態生成 docker exec wrapper script，透過各 SDK 的 binary path 設定注入（Claude `claudePath`、Gemini `GeminiAgentOptions.setExecutablePath()`、Codex `CodexAgentOptions.builder().executablePath()`）。完整重用 agent-client 0.12.2 的 `AgentModel` / `StreamingAgentModel` / `AgentResponse` 型別——零自建 AgentModel、零自建輸出解析。`StubContainerizedAgentModelFactory` 提供測試用存根（AC-1 ✅）。`ContainerizedAgentModelIT` 驗證對 grimo-runtime 容器的真實呼叫與 CLI 缺失錯誤處理（AC-2、AC-3，編譯通過待 Docker 驗證）。**附帶修正：** `cli` 模組 `allowedDependencies` 從 `{}` 改為 `{ "core" }`（Modulith 2.0.5 行為：即使 OPEN 模組仍需顯式宣告）；architecture.md、development-standards.md §13 同步修正此 drift。**里程碑 M2 完成。** 詳見 `docs/grimo/specs/archive/2026-04-18-S005-containerized-agent-cli-adapter.md` 第 7 節。
